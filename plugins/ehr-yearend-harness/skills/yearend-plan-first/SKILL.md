# Yearend Plan-First

연말정산(yjungsan) 도메인의 **변경 작업**(코드 수정·DDL·hotfix·개정세법 반영)에 대해
*무조건* plan-first 흐름을 강제하는 정책 스킬이다. 별도 동작을 수행하지 않는다 —
변경을 시작하기 전에 거쳐야 할 절차를 정의한다.

---

## 목적

연말정산 도메인은 마감 체인·세무 신고 등 **회복이 어려운 영향**이 많아, AI 가 자동으로
코드/DDL 을 변경하는 것을 막는다. 또 사용자가 이미 해결된 문제를 다시 고치려 할 때
**헛수고와 회귀 리스크**를 방지한다.

---

## 사용 시점 (이 스킬이 *발동*해야 하는 요청)

다음과 같은 **변경 작업** 요청을 받았을 때 즉시 발동.

- "출산지원금 비과세 전환 반영해줘"
- "TCPN843 에 SP_NEW_DEDUCT_CD 컬럼 추가해줘"
- "PKG_CPN_YEA_2026_CALC 의 버그 좀 고쳐줘"
- "어제 마감한 사원 결과 이상한 거 hotfix 해줘"
- "이 화면 저장 로직 수정해줘"

코드/DDL/스키마/매퍼 등 **소스 자체를 변경**하는 모든 요청이 대상.

## 사용하지 말 시점

- 단순 조회·도메인 질의 → `yearend-domain-map` 스킬
- 영향 범위 분석만 → `yearend-chain-tracer` 스킬
- 서술형·복합 조사 → `yearend-investigator` 에이전트 (이 에이전트도 본 정책을 내부적으로 따름)
- DB 직접 쓰기·DDL 실행 → 본 플러그인 범위 밖 (db-read-only 훅이 차단)

---

## 처리 절차 (5단계)

### Step 0. 현 상태 확인 (가장 먼저)

요청 사항이 **이미 코드/DB 에 반영되어 있는지 먼저 확인**한다.

- `yearend-chain-tracer` 또는 `yearend-domain-map` 으로 관련 파일·테이블·패키지·매퍼 위치를 식별
- 해당 위치의 현재 소스를 `Read` 또는 `Grep` 으로 **직접 열어 확인**
- DB 스키마/마감 체인은 `references/yjungsan-tables.md`, `references/yjungsan-close-chain.md` 와 비교

#### Case A — 이미 반영되어 있음 (또는 그렇게 추정됨)

**수정하지 않는다.** 다음 형식으로 보고하고 종료.

```
[현 상태]
요청 사항은 이미 <파일경로:line> 에 <방식> 으로 반영되어 있습니다.

  근거 코드:
  ```
  <해당 라인 인용>
  ```

[추정 원인]
보고된 증상은 위 변경 자체가 아니라 <별도 원인 후보 X> 에서 발생한 것으로 보입니다.
판단 근거: <근거 1>, <근거 2>

[다음 행동 제안]
- 추가 확인이 필요한 부분: ...
- 다른 가설로 조사 진행할까요?
```

추측이 아니라 **근거(파일·라인·코드 인용)** 를 반드시 명시한다.
근거 없이 "이미 되어 있는 것 같다" 라고 말하지 않는다.

#### Case B — 반영 안 됨

다음 단계로 진행.

### Step 1. 영향 범위 분석

`yearend-chain-tracer` 로 변경의 영향 범위를 식별한다.

- 어느 화면·컨트롤러·서비스·매퍼·테이블·패키지·마감 단계가 영향받는가
- 회귀 위험(예: 마감 후 재계산 발생, 다른 모듈에서 같은 컬럼 참조)이 있는가

### Step 2. 변경 Plan 제시

다음 항목을 포함하여 사용자에게 plan 을 제시한다.

```
[변경 대상]
- <파일 A>:<line 범위> → 변경 내용 X
- <파일 B>:<line 범위> → 변경 내용 Y
- DDL: 테이블 Z 에 컬럼 W 추가 (사용자가 직접 실행 — 본 플러그인은 DDL 자동 실행 X)

[리스크]
- ...

[검증 방법]
- ...

이대로 진행할까요? (Y / N / 수정 요청)
```

### Step 3. 사용자 명시적 승인

다음 중 하나가 도착할 때까지 **`Edit`/`Write`/`MultiEdit` 도구를 절대 호출하지 않는다**.

- 명시적 승인: `Y`, `예`, `진행`, `그대로`, `OK`, `go` 등
- 수정 요청: "X 부분만 빼고 진행" → plan 갱신 후 다시 Step 2 로

승인 메시지가 모호하면 (예: "음...") 다시 묻는다. **추측으로 진행 금지.**

### Step 4. 수정 + 검증

승인된 plan 그대로 `Edit`/`Write` 호출. 수정 후:

- 변경된 파일·라인 요약 보고
- 검증 가이드 ("이걸 직접 테스트해 주세요: ...")
- 회귀 가능 영역 재확인 가이드

---

## 정책 우회

본 정책을 우회해야 할 *진짜 비상 상황*(예: 1초 내 운영 hotfix)에서는 사용자가 다음 중 하나를 명시해야 한다.

- 직접 `Edit`/`Write` 도구를 호출
- 메시지에 `[plan-first 우회]` 같은 명시적 지시 포함

그런 명시적 지시가 **없는 한** 본 절차를 따른다. 사용자가 단순히 "급해" 라고 말한 정도로는 우회하지 않는다.

## 다른 yearend 스킬과의 관계

| 요청 유형 | 발동 스킬 | 본 스킬과의 관계 |
|----------|----------|------------------|
| 도메인 질의 ("TCPN843 이 뭐야?") | `yearend-domain-map` | 본 스킬 발동 X |
| 영향 분석 ("이 컬럼 고치면 어디 영향?") | `yearend-chain-tracer` | 본 스킬 발동 X (분석만) |
| 서술형·복합 조사 ("개정세법 영향도 봐줘") | `yearend-investigator` agent | 그 agent 가 *내부적으로* 본 정책을 따름 |
| 변경 작업 ("고쳐줘", "반영해줘") | **`yearend-plan-first`** ← 여기 | Step 0 → 1 → 2 → 3 → 4 |
