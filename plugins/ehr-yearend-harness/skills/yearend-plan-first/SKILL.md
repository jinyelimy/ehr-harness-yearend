# Yearend Plan-First

연말정산(yjungsan) 도메인의 **변경 작업**(코드 수정·DDL·hotfix·개정세법 반영)에 대해
*무조건* plan-first 흐름을 강제하는 정책 스킬이다. 별도 동작을 수행하지 않는다 —
변경을 시작하기 전에 거쳐야 할 절차를 정의한다.

---

## 목적

연말정산 도메인은 마감 체인·세무 신고 등 **회복이 어려운 영향**이 많아, AI 가 자동으로
코드/DDL 을 변경하는 것을 막는다. 또 사용자가 이미 해결된 문제를 다시 고치려 할 때
**헛수고와 회귀 리스크**를 방지한다.

---

## 사용 시점 (이 스킬이 *발동*해야 하는 요청)

다음과 같은 **변경 작업** 요청을 받았을 때 즉시 발동.

- "출산지원금 비과세 전환 반영해줘"
- "TCPN843 에 SP_NEW_DEDUCT_CD 컬럼 추가해줘"
- "PKG_CPN_YEA_2026_CALC 의 버그 좀 고쳐줘"
- "어제 마감한 사원 결과 이상한 거 hotfix 해줘"
- "이 화면 저장 로직 수정해줘"

코드/DDL/스키마/매퍼 등 **소스 자체를 변경**하는 모든 요청이 대상.

## 사용하지 말 시점

- 단순 조회·도메인 질의 → `yearend-domain-map` 스킬
- 영향 범위 분석만 → `yearend-chain-tracer` 스킬
- 서술형·복합 조사 → `yearend-investigator` 에이전트 (이 에이전트도 본 정책을 내부적으로 따름)
- DB 직접 쓰기·DDL 실행 → 본 플러그인 범위 밖 (db-read-only 훅이 차단)

---

## 처리 절차 (5단계)

### Step 0. 현 상태 확인 (가장 먼저)

요청 사항이 **이미 코드/DB 에 반영되어 있는지** 단계적으로 검사한다. 표면 키워드 매칭에 그치면 이미 반영된 변경분을 놓치므로, 아래 5개 sub-step 을 **모두** 거친 뒤에 판정한다.

#### Step 0-A. 변경 이력 우선 확인 (git)

가장 먼저 git 이력을 본다.

- `git log --all -- <관련 경로>` 로 최근 커밋 N개 확인
- `git log --all --grep=<키워드>` 로 메시지에 키워드가 포함된 커밋 검색
- 매칭되는 최근 커밋이 있으면 **그 커밋의 diff 를 정독** (`git show <hash>`)
- 결과는 `[커밋 hash | YYYY-MM-DD | 메시지]` 형태로 메모해 둠

> 사용자 환경은 yearend 도메인이라 **최근 6개월** 정도가 보통 의미 있는 시간 창. 커밋 메시지에 한국어가 자주 들어가므로 한국어 키워드도 같이 검색.

#### Step 0-B. 도메인 동의어 확장

사용자가 쓴 표면 표현(예: "사업장코드")을 [`references/yjungsan-glossary.md`](../../references/yjungsan-glossary.md) 의 동의어 사전으로 확장한다.

예: "사업장코드" → `BUSINESS_PLACE_CD`, `BP_CD`, `ENTER_CD`, `F_COM_GET_BP_CD`

확장된 키워드 **전부** 로 `Grep` / `Glob` / `git log --grep` 시도. 한 키워드 미스로 이미 반영된 변경분을 놓치는 것을 방지.

> 사전에 없는 새 키워드를 발견하면 `yjungsan-glossary.md` 에 추가하자고 사용자에게 한 줄 제안.

#### Step 0-C. 코드 정독 + 변경 흔적 grep

발견된 파일에서 **변경 표기 패턴**을 별도로 grep 한다 (사전의 "변경 흔적 패턴" 표 참고).

대표 패턴:
- 일자 표기: `(YYYY.MM.DD)`, `// YYYY-MM-DD`, `<!-- YYYY/MM/DD`
- 변경 주석: `// 추가`, `// 변경`, `// 수정`, `--- 변경`, `// FIXME`, `// TODO`
- 신규 컬럼·필드 흔적

매칭이 있으면 그 **시점·내용·라인 번호** 를 출력에 인용. 단순히 "변경 흔적 있음" 이 아니라 *어디에·언제·무엇이* 인용한다.

#### Step 0-D. 트리거 단어 감지

사용자 메시지에 다음 단어가 있으면 **부분 반영 가설을 우선** 검토한다 (사전의 "트리거 단어" 표 참고).

> "추가", "개선", "보강", "이미", "수정한 후에", "이어서", "다음 단계로", "다시"

이 단어들은 본인이 일부 작업을 끝낸 뒤의 *추가 요청* 일 가능성이 높다. 트리거 단어 감지 시 Step 0-A ~ 0-C 검사를 더 깊게 (예: 검색 키워드 더 확장, 최근 커밋 더 멀리).

#### Step 0-E. 판정

검사 결과를 다음 셋 중 하나로 분류:

| 변경 흔적 | 분기 |
|----------|------|
| **0% — 어떤 흔적도 없음** | Case B (반영 안 됨) 로 진행 |
| **부분 — 일부만 반영, 또는 모호** | Case A 가설을 사용자에게 **먼저 확인**: "이런 흔적을 찾았는데 이게 원하시는 변경의 일부 아닌가요?" 멋대로 Case B 로 점프 금지 |
| **충분 — 핵심 변경이 이미 적용됨** | Case A (이미 반영됨) 로 보고 |

#### Case A — 이미 반영되어 있음 (또는 그렇게 추정됨)

**수정하지 않는다.** 다음 형식으로 보고하고 종료. 모든 결론에 **근거(파일·라인·커밋)** 를 인용.

```
[현 상태]
요청 사항은 이미 <파일경로:line> 에 <방식> 으로 반영되어 있습니다.
  - 커밋: <hash> (<YYYY-MM-DD> "<커밋 메시지>")
  - 관련 라인:
    ```<언어>
    <해당 라인 인용>
    ```

[확인한 변경 이력]
- <커밋 hash 1> | <YYYY-MM-DD> | <메시지>
- <커밋 hash 2> | <YYYY-MM-DD> | <메시지>

[추정 원인 — 보고된 증상이 아직 풀리지 않았다면]
보고된 증상은 위 변경 자체가 아니라 <별도 원인 후보 X> 에서 발생한 것으로 보입니다.
판단 근거: <근거 1>, <근거 2>

[다음 행동 제안]
- 추가 확인이 필요한 부분: ...
- 다른 가설로 조사 진행할까요?
```

추측이 아니라 **근거(파일·라인·코드 인용·커밋 hash·날짜)** 를 반드시 명시한다.
근거 없이 "이미 되어 있는 것 같다" 라고 말하지 않는다.

#### Case B — 반영 안 됨

다음 단계로 진행.

### Step 1. 영향 범위 분석

`yearend-chain-tracer` 로 변경의 영향 범위를 식별한다.

- 어느 화면·컨트롤러·서비스·매퍼·테이블·패키지·마감 단계가 영향받는가
- 회귀 위험(예: 마감 후 재계산 발생, 다른 모듈에서 같은 컬럼 참조)이 있는가

### Step 2. 변경 Plan 제시

다음 항목을 포함하여 사용자에게 plan 을 제시한다.

```
[변경 대상]
- <파일 A>:<line 범위> → 변경 내용 X
- <파일 B>:<line 범위> → 변경 내용 Y
- DDL: 테이블 Z 에 컬럼 W 추가 (사용자가 직접 실행 — 본 플러그인은 DDL 자동 실행 X)

[리스크]
- ...

[검증 방법]
- ...

이대로 진행할까요? (Y / N / 수정 요청)
```

### Step 3. 사용자 명시적 승인

다음 중 하나가 도착할 때까지 **`Edit`/`Write`/`MultiEdit` 도구를 절대 호출하지 않는다**.

- 명시적 승인: `Y`, `예`, `진행`, `그대로`, `OK`, `go` 등
- 수정 요청: "X 부분만 빼고 진행" → plan 갱신 후 다시 Step 2 로

승인 메시지가 모호하면 (예: "음...") 다시 묻는다. **추측으로 진행 금지.**

### Step 4. 수정 + 검증

승인된 plan 그대로 `Edit`/`Write` 호출. 수정 후:

- 변경된 파일·라인 요약 보고
- 검증 가이드 ("이걸 직접 테스트해 주세요: ...")
- 회귀 가능 영역 재확인 가이드

---

## 정책 우회

본 정책을 우회해야 할 *진짜 비상 상황*(예: 1초 내 운영 hotfix)에서는 사용자가 다음 중 하나를 명시해야 한다.

- 직접 `Edit`/`Write` 도구를 호출
- 메시지에 `[plan-first 우회]` 같은 명시적 지시 포함

그런 명시적 지시가 **없는 한** 본 절차를 따른다. 사용자가 단순히 "급해" 라고 말한 정도로는 우회하지 않는다.

## 다른 yearend 스킬과의 관계

| 요청 유형 | 발동 스킬 | 본 스킬과의 관계 |
|----------|----------|------------------|
| 도메인 질의 ("TCPN843 이 뭐야?") | `yearend-domain-map` | 본 스킬 발동 X |
| 영향 분석 ("이 컬럼 고치면 어디 영향?") | `yearend-chain-tracer` | 본 스킬 발동 X (분석만) |
| 서술형·복합 조사 ("개정세법 영향도 봐줘") | `yearend-investigator` agent | 그 agent 가 *내부적으로* 본 정책을 따름 |
| 변경 작업 ("고쳐줘", "반영해줘") | **`yearend-plan-first`** ← 여기 | Step 0 → 1 → 2 → 3 → 4 |
